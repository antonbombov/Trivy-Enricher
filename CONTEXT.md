## 1. Общее описание
Проект Trivy Enricher (также упоминается как tve) предназначен для автоматического обогащения отчетов уязвимостей, сгенерированных инструментом Trivy, дополнительной информацией об эксплойтах, метриках EPSS, статусе CISA KEV, приоритетах исправления и других данных, получаемых с помощью SploitScan. Результатом являются расширенные JSON-отчеты, интерактивные HTML-отчеты с фильтрацией и поиском, а также Excel-отчеты. Проект использует кэширование для ускорения повторных запусков и многопроцессную обработку для параллельного сканирования CVE.

## 2. Основные возможности
Обогащение отчетов Trivy – добавление в каждую уязвимость блока sploitscan с данными от SploitScan.

Параллельная обработка CVE – использование ProcessPoolExecutor для одновременного запуска нескольких экземпляров SploitScan, что значительно ускоряет работу при большом количестве уникальных CVE.

Кэширование результатов SploitScan – JSON-файлы с данными по каждой CVE сохраняются в отдельной папке и при повторном запуске используются повторно (если не устарели).

Кэширование Tailwind CSS – скачивание tailwind.js с CDN один раз и сохранение в локальный кэш для офлайн-генерации HTML.

Очистка устаревшего кэша – автоматическое удаление файлов старше заданного количества дней (по умолчанию 30).

Генерация интерактивных HTML-отчетов – с фильтрацией по CVE, пакету, приоритету, severity, EPSS, статусу, наличию эксплойтов, CISA KEV, а также с древовидной навигацией по секциям и пакетам.

Генерация Excel-отчетов – плоская таблица всех вхождений уязвимостей с колонками: источник, путь, пакет, версия, CVE, уровень критичности, срок устранения, статус (с выпадающим списком), комментарий. Срок устранения рассчитывается динамически на основе severity и наличия эксплойтов.

Дедупликация в HTML – группировка одинаковых уязвимостей (CVE + пакет + версия) для компактного отображения, с сохранением всех путей вхождений.

Логирование вызовов SploitScan – для каждой попытки сканирования CVE создается отдельный лог-файл в папке logs внутри выходной директории.

**Управление через аргументы командной строки**:
- `-html` – генерация только HTML отчетов
- `-excel` – генерация только Excel отчетов (оба листа)
- `-skip-enrich` / `-se` – пропустить обогащение SploitScan, использовать исходный отчет Trivy (полезно для быстрой генерации отчетов из уже имеющихся данных)
- `-h` / `-help` – показ справки
- Без аргументов – показ справки (обязательно указание режима)

## 3. Архитектура проекта

`main.py` – точка входа, управление общим процессом: загрузка конфигурации, очистка логов, поиск отчетов Trivy, запуск обогащения и последующая генерация отчетов.

`config_manager.py` – загрузка и сохранение конфигурации из config.json, проверка путей, создание необходимых директорий.

`enrichment_core.py` – ядро обогащения: извлечение CVE из отчета Trivy, проверка кэша, организация параллельной обработки, объединение результатов и сохранение обогащенного JSON.

`parallel_processor.py` – вспомогательный модуль для расчета оптимального количества воркеров и выполнения одного CVE в отдельном процессе.

`sploitscan_runner.py` – запуск SploitScan для конкретной CVE, обработка таймаутов, запись логов, перемещение полученного JSON в кэш.

`sploitscan_parser.py` – парсинг JSON-файла от SploitScan и возврат данных (без изменений).

`trivy_parser.py` – извлечение списка CVE из исходного отчета Trivy.

`trivy_html_reporter.py` – генерация HTML-отчета из обогащенного JSON, включая сбор статистики, группировку, формирование интерактивных элементов.

`html_templates.py` – содержит базовый HTML-шаблон, CSS-стили и JavaScript-код для интерактивности.

`cdn_cache_manager.py` – управление кэшированием Tailwind JS: скачивание, чтение из кэша, статистика.

`cache_cleaner.py` – очистка старых файлов в кэше SploitScan на основе cache_max_days.

`argument_parser.py` – выделенный модуль для парсинга аргументов командной строки.

`excel_reporter.py` – объединенный модуль для генерации Excel отчетов:
- `generate_excel_report()` – создает Excel файл с листами
- `add_sca_sheet()` – добавляет лист SCA Анализ
- `add_ptai_sheet()` – добавляет лист PTAI Анализ (при совпадении имен)

`ptai_processor.py` – модуль для обработки PTAI отчетов на базе lxml:
- Класс `PTAIParser` – парсинг HTML с использованием XPath
- `prepare_ptai_excel_data()` – подготовка данных для Excel
- `find_ptai_report_for_trivy()` – поиск PTAI отчета

`excel_styles.py` – централизованное хранилище стилей для Excel.

## 4. Описание модулей
### 4.1 config_manager.py
Функция `load_config()` читает config.json из той же директории, что и скрипт, и дополняет недостающие поля значениями по умолчанию. Особое внимание уделяется обработке max_workers (может быть null или числом). Возвращает словарь конфигурации.

`get_sploitscan_path()` определяет, является ли sploitscan_path командой в PATH или путем к конкретному .py файлу. Возвращает строку с путем или None, если не найдено.

`setup_directories()` создает директории для кэша и выходных данных (если их нет). Директория для входных отчетов (scan_directory) не создается, так как предполагается, что она уже существует.

### 4.2 cache_cleaner.py
`cleanup_old_cache()` – удаляет все .json файлы в папке кэша, чей возраст превышает cache_max_days (в секундах). Возвращает количество удаленных файлов.

`get_cache_stats()` – возвращает общее количество файлов в кэше и максимальный возраст.

### 4.3 cdn_cache_manager.py
`get_tailwind_js(cache_dir)` – проверяет наличие файла tailwind.js в подпапке cdn внутри cache_dir. Если файл есть – читает его и возвращает содержимое. Если нет – скачивает с https://cdn.tailwindcss.com/ и сохраняет.

`get_cdn_cache_stats()` – возвращает статус кэша Tailwind.

`clear_cdn_cache()` – удаляет закэшированный Tailwind.

### 4.4 trivy_parser.py
`extract_cves_from_trivy(trivy_report_path)` – открывает JSON-отчет Trivy, проходит по всем результатам и собирает уникальные идентификаторы уязвимостей, начинающиеся с CVE-. Возвращает отсортированный список.

### 4.5 sploitscan_runner.py
`find_existing_json(cve_id, target_dir)` – ищет файл с именем, содержащим cve_id, в целевой директории.

`setup_sploitscan_logging(cve_id, attempt)` – создает лог-файл в output_dir/logs с именем sploitscan_{cve_id}_attempt{attempt}.log.

`read_output(process, log_file)` – поток, читающий stdout процесса в реальном времени и пишущий в лог.

`run_sploitscan(cve_id, target_dir, attempt=1)` – запускает SploitScan с параметрами:
- Для .py файла: python <путь> -e json <cve_id>.
- Для команды sploitscan: sploitscan -e json <cve_id>.

Рабочая директория – родительская папка скрипта или корень проекта.

Устанавливается таймаут из конфига (по умолчанию 60 сек).

В случае таймаута процесс принудительно завершается.

При успешном завершении ищет созданный JSON файл (по маске *{cve_id}*.json) и перемещает его в target_dir (кэш).

Возвращает словарь с status (success, cached, failed), путем к файлу, временем выполнения и номером попытки.

### 4.6 sploitscan_parser.py
`parse_sploitscan_data(json_file)` – загружает JSON и возвращает его содержимое как есть. Если файл содержит список, берет первый элемент.

### 4.7 parallel_processor.py
`process_single_cve(args)` – принимает кортеж (cve_id, target_dir, attempt), вызывает run_sploitscan и парсит результат. Возвращает (cve_id, результат), где результат содержит данные и статус.

`calculate_optimal_workers(total_cves, config_max_workers=None)` – определяет число параллельных процессов:

Если в конфиге задано число – использует его.

Иначе автоматически: при <=10 CVE – не более 3, <=30 – до 5, иначе до 7, но не превышая количество ядер CPU.

### 4.8 enrichment_core.py
`enrich_trivy_report(trivy_report_path, output_dir)` – основной процесс обогащения:
- Загружает конфиг.
- Читает исходный Trivy-отчет.
- Получает список CVE.
- Разделяет CVE на уже закэшированные и требующие сканирования.
- Рассчитывает оптимальное количество воркеров.
- Запускает параллельную обработку для некэшированных CVE (первая попытка).
- Для упавших CVE делает повторную попытку (второй проход).
- Собирает результаты из кэша и сканирования.
- Для CVE, которые не удалось обработать после двух попыток, записывает заглушку {"error": "Не удалось получить данные после 2 попыток"}.
- Обогащает оригинальный отчет, добавляя поле sploitscan в каждую уязвимость.
- Сохраняет обогащенный JSON в `output_dir` с суффиксом _enriched.json.
- Возвращает путь к сохраненному файлу.

### 4.9 trivy_html_reporter.py
`generate_trivy_html_report(enriched_trivy_path, output_dir, cache_dir)` – основной генератор HTML:
- Загружает обогащенный JSON.
- Получает Tailwind JS из кэша/CDN.
- Вызывает `collect_statistics_and_group_data()` для сбора статистики и группировки уязвимостей.
- Генерирует HTML, используя шаблоны из html_templates.py и встраивая Tailwind JS в нужное место.
- Сохраняет файл с суффиксом _report.html.

`collect_statistics_and_group_data(trivy_data)` – подсчитывает:
- Общее количество уязвимостей (с дублями и уникальные).
- Распределение по severity (Critical, High, Medium, Low, Unknown) с указанием количества уязвимостей, имеющих эксплойты.
- Количество CVE с эксплойтами и в CISA KEV.
- Группирует уязвимости по секциям (Type/Target) и пакетам.

`group_vulnerabilities_by_unique_key(vulnerabilities)` – дедупликация для HTML: группирует по CVE + PURL (или CVE + PkgName + Version). Для каждой группы сохраняет множество путей, статусов, фиксов и общее количество вхождений.

`generate_vulnerability_card(vuln_data)` – создает HTML-код карточки уязвимости с учетом дублей (отображает либо один путь, либо список всех путей).

`get_exploit_data(sploitscan)` – извлекает структурированные данные об эксплойтах из разных источников (GitHub, ExploitDB, NVD, Metasploit, VulnCheck).

`generate_exploits_section(...)` – формирует блок с эксплойтами, если они есть.

`generate_sidebar(...)` – создает левую панель с фильтрами и древовидной навигацией по секциям/пакетам.

### 4.10 html_templates.py
Содержит три основные функции:

`get_base_html()` – базовый HTML-шаблон с плейсхолдерами для контента, CSS, JS и Tailwind.

`get_css_styles()` – строка с CSS-стилями (в основном для печати, цветов и скрытия скроллбаров).

`get_javascript()` – строка с JS-кодом для темы, фильтрации и навигации.

### 4.11 argument_parser.py
`parse_arguments()` – парсит аргументы -html, -excel, -h. При отсутствии аргументов показывает справку.

`get_report_types()` – возвращает кортеж (generate_html, generate_excel).

`get_report_types(args)` – возвращает кортеж `(generate_html, generate_excel, skip_enrich)` для удобного использования.

### 4.12 ptai_processor.py
Класс `PTAIParser` – парсинг HTML отчетов PTAI с использованием lxml и XPath:

```python
# Основные XPath выражения
root_tables = tree.xpath("//table[@class='vulnerability-root-table']")
confirmed = tbl.xpath(".//div[@class='vulnerability-statuses']//i[@title='Подтверждена']")
rejected = tbl.xpath(".//div[@class='vulnerability-statuses']//i[@title='Опровергнута']")
```
Парсер находит:
- Подтвержденные уязвимости (иконка с title='Подтверждена')
- Опровергнутые уязвимости (иконка с title='Опровергнута')
- ID, файл, тип, CWE, комментарий

### 4.13 excel_reporter.py
`generate_excel_report(enriched_trivy_path, output_dir, ptai_html_path=None)` – создает Excel с листами SCA Анализ и (опционально) PTAI Анализ.

### 4.14 excel_styles.py
Централизованное хранилище стилей: цвета, шрифты, границы, ширина колонок для обоих листов.

### 4.15 main.py
Выполняет:
- Загрузку конфигурации и создание директорий.
- Вывод статистики кэша (SploitScan и CDN).
- Очистку устаревшего кэша.
- Очистку папки с логами (удаление всех .log файлов).
- Поиск всех .json файлов в scan_directory, исключая _enriched.json и config.json.

Для каждого найденного отчета:
- Запуск enrich_trivy_report.
- Если успешно – генерация HTML и Excel отчетов с замерами времени.
- Вывод итогов.

## 5. Конфигурация (config.json)
Поля:
- `sploitscan_path` – путь к исполняемому файлу SploitScan (или .py скрипту). Если указано "sploitscan", то ищется в PATH.
- `scan_directory` – абсолютный путь к папке с исходными отчетами Trivy. Внутри этой папки ожидается подпапка **PTAI** с HTML отчетами.
- `cache_directory` – абсолютный путь для хранения кэша результатов SploitScan.
- `output_directory` – абсолютный путь для сохранения обогащенных JSON, HTML, Excel и логов.
- `cache_max_days` – максимальный возраст файлов в кэше (в днях).
- `max_workers` – ограничение на количество параллельных процессов (если null – автоопределение).
- `timeout` – таймаут на выполнение одного вызова SploitScan (сек).
- `project_version` – версия проекта (выводится в хежеое подвале HTML).

## 6. Процесс работы (пошагово)
Пользователь помещает JSON-отчет Trivy (строго UTF-8) в папку scan_directory.

Запускается main.py с указанием режима:
- `python main.py -html` – только HTML отчет (с обогащением)
- `python main.py -excel` – только Excel отчет (с обогащением)
- `python main.py -html -excel` – оба отчета (с обогащением)
- `python main.py -html -skip-enrich` – только HTML отчет без обогащения (используется исходный Trivy JSON)
- `python main.py -excel -skip-enrich` – только Excel отчет без обогащения
- `python main.py -html -excel -skip-enrich` – оба отчета без обогащения

**Режим без обогащения (`-skip-enrich`)**:
- Пропускает вызов SploitScan и обогащение отчета trivy
- Использует исходный JSON отчет Trivy как есть
- Ускоряет генерацию отчетов, если обогащение не требуется

Происходит очистка устаревшего кэша и логов.

Для каждого отчета:
- Извлекаются все CVE.
- Проверяется наличие в кэше: для закэшированных CVE данные берутся сразу.
- Оставшиеся CVE обрабатываются параллельно (с первой попыткой).
- Неудачные CVE повторяются во второй попытке (снова параллельно).
- Формируется обогащенный JSON и сохраняется.
- Генерируется HTML-отчет (если указан -html): статистика, группировка, интерактивность.

**Генерация Excel отчета** (если указан -excel):
- Загрузка обогащенного JSON
- Поиск соответствующего PTAI отчета в папке `scan_directory/PTAI/`
- При совпадении имен – создается Excel с двумя листами
- При несовпадении – создается только лист SCA Анализ

Все выходные файлы сохраняются в output_directory.

## 7. Особенности и важные нюансы
### 7.1 Обработка ошибок SploitScan
Если SploitScan завершается с ошибкой или по таймауту, результат помечается как failed. После повторной попытки, если снова неудача, в обогащенный отчет добавляется {"error": "Не удалось получить данные после 2 попыток"}.

В HTML-отчете для таких CVE:
- priority = 'Not scanned', epss_score = 'Not scanned', cisa_status = 'Not scanned'.
- Секция с эксплойтами отсутствует.
- Фильтр "Not scanned by SploitScan" позволяет показать только такие уязвимости.

### 7.2 Дедупликация в HTML
Группировка выполняется по комбинации VulnerabilityID и PkgIdentifier.PURL (если есть), иначе по PkgName + InstalledVersion.

Для каждой группы сохраняются все уникальные пути (PkgPath), статусы и фиксированные версии.

В карточке отображается общее количество вхождений. Если вхождений >1, показывается раскрывающийся блок со списком всех путей.

Статистика в карточках severity и эксплойтов учитывает только уникальные уязвимости.

### 7.3 Логирование SploitScan
Каждый вызов SploitScan логируется в отдельный файл в output_dir/logs/.

Имя файла: sploitscan_{cve_id}_attempt{номер}.log.

В лог пишется команда, рабочая директория, время начала, весь вывод процесса, код возврата и время окончания.

Это помогает отлаживать проблемы со сканированием конкретных CVE.

### 7.4 Кэширование Tailwind
Tailwind JS скачивается один раз и сохраняется в cache_dir/cdn/tailwind.js.

При генерации HTML содержимое этого файла вставляется внутрь тега <script>.

Если скачивание не удалось, используется CDN-ссылка как запасной вариант.

### 7.5 Расчет сроков устранения в Excel
Используется dateutil.relativedelta для прибавления месяцев.

Базовые сроки: Critical – 6 мес, High – 9, остальные – 12.

Если есть публичные эксплойты, срок сокращается на 3 месяца (но не менее 1 месяца).

Дата отсчитывается от текущей даты на момент генерации отчета.

### 7.6 Фильтрация в HTML
Реализована на чистом JavaScript без сторонних библиотек.

Фильтры: поиск по CVE и пакету, приоритет (A+, A, B, C, D, Not scanned), severity, статус (fixed, affected и др.), EPSS (>= введенного процента), CISA KEV, наличие эксплойтов, "не отсканировано".

При применении фильтров карточки скрываются, также скрываются целые пакеты и секции, если в них не осталось видимых карточек.

Счетчик видимых карточек появляется при активных фильтрах.

### 7.7 Навигация по секциям и пакетам
В левой панели строится дерево секций.

Каждая секция имеет кнопку для раскрытия списка пакетов внутри.

Клик по названию секции или пакета осуществляет плавный скролл к соответствующему месту в отчете.

При переходе по ссылке с хэшем автоматически раскрывается нужная секция.

### 7.8 Обработка путей (PkgPath) в JAR/WAR
Для путей вида app.jar/BOOT-INF/lib/lib.jar корневой JAR (app.jar) выделяется в отдельную колонку в Excel и отображается в карточке.

В HTML для множественных вхождений показываются полные пути, но для удобства длинные пути обрезаются до 60+60 символов с многоточием.

### 7.9 Очистка логов перед запуском
Все файлы .log в output_dir/logs удаляются при старте main.py, чтобы не накапливались.

### 7.10 Парсинг PTAI отчетов
Используется lxml с XPath выражениями. Парсер находит:
- Подтвержденные уязвимости (`title='Подтверждена'`)
- Опровергнутые уязвимости (`title='Опровергнута'`)
- Уязвимости без статуса не включаются в отчет

### 7.11 Аргументы командной строки
Обязательное указание режима работы. Без аргументов показывается справка.

### 7.12 Режим без обогащения
При использовании ключа `-skip-enrich` или `-se`:
- SploitScan не вызывается
- Кэш не проверяется и не обновляется
- Логи не очищаются (так как они не создаются)
- HTML и Excel отчеты генерируются напрямую из исходного Trivy JSON
- PTAI лист в Excel создается при наличии соответствующего HTML отчета (как и в обычном режиме)

## 8. Выходные данные
Обогащенный JSON: оригинальный отчет Trivy + поле sploitscan для каждой уязвимости.

HTML-отчет: интерактивный документ с фильтрами, статистикой, карточками и навигацией.

Excel-отчет: таблица для дальнейшей обработки (например, в DefectDojo или ручного заполнения статусов).

Логи SploitScan: отдельные файлы для каждого вызова.

## 9. Зависимости
Python 3.7+
openpyxl – для генерации Excel.
requests – для скачивания Tailwind.
python-dateutil – для расчета сроков устранения.
lxml – для парсинга PTAI HTML отчетов.

Trivy (внешний инструмент) – для получения исходных отчетов.
SploitScan (внешний инструмент) – рекомендуется модифицированная версия, которая возвращает информационные ссылки даже при отсутствии эксплойтов.

## 10. Важные замечания
Все пути в config.json должны быть абсолютными, чтобы скрипт мог работать из любого места.

Кодировка всех JSON-файлов должна быть UTF-8 (Trivy может генерировать UTF-8 with BOM, поэтому используется utf-8-sig).

При использовании Docker-контейнера монтируются три папки: input (для отчетов Trivy), results (для выходных данных) и опционально cache (для сохранения кэша между запусками).

Проект версионируется через поле project_version в конфиге, которое отображается в подвале HTML.

Для корректной работы фильтрации в HTML необходимо, чтобы у каждой карточки были data-атрибуты с CVE, пакетом, приоритетом, severity, EPSS, CISA, наличием эксплойтов, статусом и признаком сканирования. Это автоматически формируется при генерации.
